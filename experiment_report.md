# 深度学习大作业：AI 美食食谱推荐系统实验报告

## 1. 实验概述 (Overview)

### 1.1 实验背景
随着社交媒体和数字生活的普及，人们越来越习惯通过图片分享美食。然而，从一张美食图片到获取其具体的烹饪方法，往往需要繁琐的搜索过程。本实验旨在构建一个端到端的智能系统，连接“视觉感知”与“知识检索”，用户只需上传一张图片，系统即可自动识别食物类别，并推荐详细的烹饪食谱。

### 1.2 实验目标
1.  构建一个高精度的细粒度图像分类模型，能够在 101 种美食类别上达到较高的识别准确率。
2.  利用自然语言处理技术，从非结构化的海量食谱文本中检索出与视觉类别最匹配的结构化食谱。
3.  开发一个用户友好的 Web 应用，集成图像识别、食谱推荐和跨语言翻译功能。



---

## 2. 实验数据 (Datasets)

本实验涉及两个异构数据集的处理：视觉数据集 (Food-101) 和文本数据集 (RecipeNLG)。

### 2.1 视觉数据集 (Food-101)

**数据概况**: 
Food-101 是一个具有挑战性的细粒度图像分类数据集，由 ETH Zurich 发布。
*   **规模**: 包含 101,000 张真实世界的美食图片，共 101 个类别（如 `apple_pie`, `bibimbap`, `peking_duck` 等），每个类别包含 1,000 张图片。
*   **特点**: 
    *   **多样性**: 图片来自真实的社交媒体，背景复杂，包含各种拍摄角度、光照条件和摆盘方式，存在一定的噪声（如非目标食物、标签错误）。
    *   **均衡性**: 所有类别的样本数量完全一致（各 1000 张），无需处理类别不平衡问题。
    *   **挑战**: 某些类别外观极为相似（如 `filet_mignon` 和 `steak`），对模型的细粒度特征提取能力要求较高。
*   **划分**: 官方提供 `train.txt` (75,750 张) 和 `test.txt` (25,250 张)。本实验为便于调优，重新进行了 9:1 的训练/验证划分。

### 2.2 文本数据集 (RecipeNLG)

**数据概况**: 
RecipeNLG 是一个大规模的非结构化自然语言文本数据集，旨在促进食谱生成与理解的研究。
*   **规模**: 包含超过 223 万条食谱文本数据 (CSV 格式)，总大小超过 2GB。
*   **来源**: 爬取自多个主流烹饪网站（如 Food.com, AllRecipes 等），数据来源广泛且杂乱。
*   **结构**: 每条数据包含以下关键字段：
    *   `title`: 食谱名称（如 "Old Fashioned Apple Pie"）。
    *   `ingredients`: 原料列表（JSON 数组格式，如 `["apple", "sugar", "flour"]`）。
    *   `directions`: 烹饪步骤列表（JSON 数组格式）。
    *   `link` / `source`: 来源元数据。
    *   `NER`: 命名实体识别提取出的食材实体（本实验未直接使用）。
*   **问题**: 原始数据存在大量噪声，包括：
    *   **重复**: 同名或极度相似的食谱大量存在。
    *   **格式错误**: 包含非 ASCII 字符、HTML 标签残留。
    *   **不相关**: 包含大量非 Food-101 类别的食谱。

## 3. 实验内容与方法 (Experimental Content and Methods)

本实验主要包含三个核心内容模块，分别采用不同的技术方法实现：

### 3.1 细粒度图像分类 (Fine-grained Image Classification)
*   **实验内容**: 训练一个深度神经网络，使其能够区分 101 种外观相似的食物（如“排骨”与“牛排”）。
*   **方法**: **迁移学习 (Transfer Learning)**。
    *   选用 **EfficientNet-B3** 作为骨干网络（Backbone），该网络在 ImageNet 上预训练过，具备强大的特征提取能力。
    *   通过微调（Fine-tuning）网络的最后全连接层（Classification Head），使其适配 Food-101 数据集的分类任务。
    *   采用 **数据增强 (Data Augmentation)** 技术（旋转、翻转、色彩抖动）来提高模型的泛化能力。

### 3.2 跨模态食谱检索 (Cross-modal Recipe Retrieval)
*   **实验内容**: 建立视觉类别（如 "Apple Pie"）与文本食谱数据库之间的语义映射关系，实现从“标签”到“详细内容”的精准推荐。
*   **方法**: **语义向量化 (Semantic Embedding)**。
    *   使用 **Sentence-BERT** (all-MiniLM-L6-v2) 模型将食物类别名称和食谱文本映射到同一个高维语义空间。
    *   利用 **余弦相似度 (Cosine Similarity)** 计算向量间的距离，筛选出语义最匹配的食谱，并构建 Top-K 索引库。

### 3.3 全栈系统开发 (Full-stack System Development)
*   **实验内容**: 将训练好的模型和处理好的数据集成到一个 Web 应用中，提供可视化交互界面。
*   **方法**: **前后端分离架构**。
    *   **后端**: 使用 **FastAPI** (Python) 构建 RESTful API，负责模型推理、数据库查询和外部翻译接口封装。
    *   **前端**: 使用 HTML5/CSS3/JavaScript 构建用户界面，实现图片预览、异步请求和动态渲染。
    *   **翻译**: 集成 **百度翻译 API**，通过哈希签名认证和请求限流算法解决跨语言障碍。

### 3.4 真实场景验证 (Real-world Validation)
*   **实验内容**: 为了验证模型在非实验室环境下的泛化能力，构建了一个独立的测试数据集。
*   **方法**: **网络爬虫 (Web Scraping)**。
    *   编写自动化脚本，从搜索引擎（Bing Images）爬取 Food-101 的 101 个类别的图片。
    *   应用 **关键词优化策略**（如添加 "food", "dish" 或使用中文译名）来过滤非食物噪声（如将 "Hot Dog" 误识别为玩具车）。
    *   构建包含 500+ 张图片的独立测试集，对模型进行盲测。

## 4. 实验步骤与过程 (Experimental Steps and Process)

### 步骤一：数据准备与预处理 (Data Preparation)

1.  **视觉数据处理 (Food-101)**:
    *   **清洗**: 遍历 101,000 张图片，剔除损坏文件，并将所有图片转换为 RGB 格式。
    *   **增强**: 定义 PyTorch 数据变换流水线。训练集应用 `RandomHorizontalFlip` (p=0.5), `RandomRotation` (±15°), `ColorJitter` (亮度/对比度/饱和度)；验证集仅进行 `Resize(224x224)` 和 `ToTensor` 归一化。
    *   **加载**: 使用 `DataLoader` 封装数据集，Batch Size 设为 32。

2.  **文本数据处理 (RecipeNLG)**:
    *   **筛选**: 编写脚本遍历 223 万条食谱数据，基于 101 个类别关键词（及其变体）筛选出相关子集。
    *   **拼接**: 将结构化的食谱字段（标题、原料、步骤）拼接成单一的语义文本块，以便于模型理解上下文。
    *   **向量化**: 调用 Sentence-BERT 模型对筛选后的文本进行编码，生成 384 维的 `.jsonl` 嵌入文件。

### 步骤二：模型构建与训练 (Model Training)

1.  **网络初始化**:
    *   加载 `efficientnet_b3` 的 ImageNet 预训练权重。
    *   修改分类层：`model.classifier[1] = nn.Linear(num_ftrs, 101)`。

2.  **训练配置**:
    *   优化器：`AdamW` (LR=1e-4)。
    *   损失函数：`CrossEntropyLoss`。
    *   早停策略：监控验证集准确率，若 8 个 Epoch 无提升则停止。

3.  **迭代训练**:
    *   在 GPU 上进行前向传播计算 Loss，反向传播更新权重。
    *   每个 Epoch 结束后评估验证集表现，并保存最佳模型权重 `best_efficientnet_b3_food101.pth`。

### 步骤三：语义匹配与数据库构建 (Semantic Matching)

1.  **基准计算**: 计算 101 个类别名称的语义向量。
2.  **相似度排序**: 对每个类别，计算其基准向量与所有候选食谱向量的余弦相似度。
3.  **索引生成**: 保留每个类别相似度最高的 Top-10 食谱，生成最终的静态数据库 `final_recipes.json`。

### 步骤四：系统集成与接口开发 (System Integration)

1.  **后端开发 (`app.py`)**:
    *   实现模型与数据的“一次性加载”机制（Startup Event）。
    *   开发 `/predict` 接口：接收图片 -> 预处理 -> 模型推理 -> 获取类别 -> 查库 -> **去重**（解决同名食谱问题） -> 返回 Top-5 结果。
    *   开发 `/translate` 接口：封装百度翻译 API，处理签名逻辑。

2.  **前端开发 (`index.html`)**:
    *   实现拖拽上传与图片预览组件。
    *   编写 `fetch` 逻辑与后端通信。
    *   实现食谱卡片的动态渲染（支持折叠/展开）。
    *   **实现翻译限流**: 编写 `translateRecipe` 函数，利用 `await delay(1200)` 策略串行发送标题、原料、步骤的翻译请求，规避 API QPS 限制。

### 步骤五：翻译功能实现 (Translation Implementation)

1.  **后端接口 (`/translate`)**:
    *   **封装 API**: 申请百度翻译通用文本 API，获取 AppID 和 Key。
    *   **签名生成**: 根据百度官方文档，使用 MD5 算法生成签名 `sign = md5(appid + q + salt + key)`。
    *   **请求转发**: 使用 `requests` 库向百度服务器发送 HTTP POST 请求，并解析返回的 JSON 数据。

2.  **前端交互与限流**:
    *   **异步调用**: 在 `index.html` 中编写 `translateRecipe` 函数，分别对食谱的标题、原料和步骤进行翻译。
    *   **串行限流**: 针对免费版 API 的 QPS (每秒查询率) 限制，实现了 `delay(ms)` 函数。在连续发送翻译请求之间强制等待 1200ms，有效避免了 `500 Internal Server Error` 或 API 频率超限错误。

### 步骤六：爬虫验证与优化 (Crawler Validation & Optimization)

1.  **初始爬取**:
    *   开发 `scrape_images.py`，利用正则表达式从 Bing 图片搜索结果中提取图片 URL。
    *   初次测试发现部分类别（如 `hot_dog`, `fish_and_chips`）准确率为 0%，经检查发现爬取到了大量无关图片（如玩具车、卡通图）。

2.  **多轮修复策略**:
    *   **英文优化**: 将关键词修改为 "hot dog food", "fish and chips dish" 等，过滤部分噪声。
    *   **中文策略**: 针对依然识别困难的类别（如 `pad_thai`），尝试使用中文关键词（"泰式炒河粉 美食"）进行爬取，准确率显著提升。
    *   **混合回滚**: 发现部分西式菜肴（如 `garlic_bread`）用中文搜索反而效果变差，因此回滚至英文关键词。

3.  **最终测试**:
    *   构建了包含 520 张图片的清洗后测试集。
    *   编写 `evaluate_model.py` 和 `plot_accuracy.py`，生成详细的分类报告（Classification Report）和准确率可视化图表。

## 5. 实验结果 (Results)

*   **分类性能**: 训练耗时约 43 分钟，在第 16 个 Epoch 触发早停。最终模型在验证集上的准确率达到 **80.05%**。尽管训练集准确率高达 96.11%（存在一定过拟合），但该性能已足以支撑实际应用。
*   **真实场景泛化**: 在通过爬虫构建的独立测试集（520 张网络图片）上，模型最终达到了 **78.46%** 的准确率。这表明模型具有极强的鲁棒性，在非同源数据上的性能下降极小（仅下降约 1.6%）。
*   **检索质量**: 推荐系统返回的食谱与识别结果高度契合，且去重逻辑有效保证了推荐列表的多样性。
*   **交互体验**: 系统响应迅速，翻译功能稳定（无报错），成功打破了原英文食谱的语言障碍，实现了从“看图”到“做菜”的完整闭环。

## 6. 总结 (Conclusion)

本实验成功实现了一个基于深度学习的美食推荐系统。通过迁移学习解决了细粒度图像分类问题，通过语义向量匹配解决了非结构化文本检索问题，并通过前后端工程化开发实现了一个可用的 Web 应用。实验过程中，我们通过引入后端去重逻辑解决了数据冗余问题，通过前端串行限流策略解决了 API 调用限制问题，最终交付了一个功能完善、体验良好的系统。

## 7. 实验体会与心得 (Reflections and Insights)

通过本次“AI 美食食谱推荐系统”的设计与实现，我深刻体会到了理论知识与工程实践相结合的重要性，并有以下几点主要收获：

1.  **深度学习落地挑战**:
    在实验初期，我发现直接训练深层网络容易过拟合且收敛慢。采用 **迁移学习** (Transfer Learning) 策略，利用 EfficientNet-B3 的预训练权重，不仅加快了训练速度，还显著提升了模型在 Food-101 数据集上的泛化能力。这让我明白了在小样本或特定领域任务中，“站在巨人的肩膀上”是多么关键。

2.  **数据质量决定上限**:
    在处理 RecipeNLG 数据集时，面对 200 多万条杂乱的文本数据，我意识到单纯的模型调优不如高质量的数据清洗有效。通过清洗噪声数据、筛选特定类别以及使用 Sentence-BERT 进行语义向量化，极大地提升了检索的准确性。这验证了机器学习界的名言：“Garbage In, Garbage Out”。

3.  **全栈开发的系统思维**:
    从后端的 FastAPI 接口设计到前端的异步请求处理，再到第三方翻译 API 的集成，每一个环节都需要精心设计。特别是在解决百度翻译 API 的 QPS 限制时，我不得不从用户体验（快速响应）和系统稳定性（避免报错）之间寻找平衡，最终采用的前端串行限流策略是一个简单但有效的工程化解决方案。

4.  **不足与展望（面向保研/科研的进阶思考）**:
    虽然系统已具备基本功能，但仍有较大的学术探索空间：
    *   **大语言模型 (LLM) 与强化学习的融合**：目前的食谱推荐基于静态的 Sentence-BERT 语义检索，返回的食谱往往缺乏个性化，且机器翻译的文本生硬。未来我计划引入 **LLM (如 LLaMA 或 ChatGLM)** 作为生成器，并结合 **基于人类反馈的强化学习 (RLHF)** 或 **DPO (直接偏好优化)** 技术。通过设计奖励模型（Reward Model），根据“食谱的可操作性”、“食材的易获取性”以及“营养均衡度”对 LLM 生成的食谱进行打分，从而微调大模型，使其不仅能生成食谱，还能生成“符合中国家庭厨房条件、健康且美味”的高质量定制化食谱。
    *   **跨模态细粒度对齐**：目前的流水线是“图片 -> 标签 -> 文本”，丢失了大量视觉细节。未来可尝试使用 CLIP 等多模态架构，将食物图片的细粒度特征（如配菜、烹饪方式）直接与食谱文本空间对齐，实现更精准的 Image-to-Text 检索。

总的来说，这次大作业不仅锻炼了我的编码能力，更培养了我解决复杂问题的系统性思维。这为我未来在人工智能领域的深入研究打下了坚实的基础。
